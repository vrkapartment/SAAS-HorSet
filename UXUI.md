# ✨ Premium UX/UI Design System & AI Guidelines

เอกสารนี้คือคู่มือการออกแบบ UI และ UX ระดับสูงสุด (Premium SaaS Standard) เพื่อให้ระบบดูแพง ทันสมัย ใช้งานง่าย และน่าเชื่อถือ โปรดใช้กฎเหล่านี้เป็นบรรทัดฐานในการเขียนโค้ด UI Components, Layouts และ Interactions ทั้งหมดในโปรเจกต์นี้

---

## 🎨 PART 1: UI (User Interface) - กฎการสร้างหน้าตาให้ "ดูแพงและพรีเมียม"

ความพรีเมียมไม่ได้เกิดจากการใส่กราฟิกเยอะๆ แต่เกิดจาก **"ความสะอาดตา (Minimalism), พื้นที่ว่าง (Whitespace), และความละเอียดอ่อน (Attention to detail)"**

### 1. Color Palette (การเลือกใช้สี)
*   **กฎเหล็ก:** ใช้สีโทนกลาง (Neutral Colors) เป็นพื้นที่ 80% ของหน้าจอ เพื่อให้แอปดูสะอาดและโปร่งตา
*   **Neutral Base:** ใช้โทนสี `Slate` หรือ `Zinc` แทนสีดำ/เทาปกติ เพื่อให้ดูมีความนุ่มนวล
    *   พื้นหลัง (Light): `bg-slate-50` หรือ `bg-gray-50` (หลีกเลี่ยงพื้นขาวจั๊วะ `bg-white` ในส่วนของ Background นอก Card)
    *   พื้นหลัง (Dark): `bg-slate-900` หรือ `bg-[#0f172a]` (หลีกเลี่ยงสีดำสนิท `#000000`)
*   **Primary Accent:** ใช้สีหลัก (เช่น Blue, Indigo หรือ Emerald) เฉพาะจุดที่ต้องการดึงสายตา หรือปุ่ม Call to Action เท่านั้น

### 2. Typography (การจัดวางตัวอักษร)
*   **กฎเหล็ก:** ข้อความต้องอ่านง่ายและมีการไล่ระดับความสำคัญ (Hierarchy) ชัดเจน
*   **Headings:** หัวข้อต้องตัวหนาและกระชับ ใช้ Tracking (ระยะห่างตัวอักษร) ที่ติดกันเล็กน้อยให้ดูโมเดิร์น
    *   Tailwind: `text-2xl font-semibold tracking-tight text-slate-900 dark:text-white`
*   **Body & Muted Text:** ข้อมูลรองห้ามใช้สีดำเข้ม ให้ใช้สีเทาเพื่อลดการแย่งสายตา
    *   Tailwind: `text-sm text-slate-500 dark:text-slate-400 leading-relaxed`

### 3. Whitespace & Layout (พื้นที่ว่างและการจัดวาง)
*   **กฎเหล็ก:** "หายใจสะดวก" อย่าจับองค์ประกอบยัดให้ชิดกัน การมีพื้นที่ว่างเยอะๆ คือคีย์หลักของแอปที่ดูแพง
*   **Spacing:** ใช้ Padding และ Margin ให้กว้างขึ้นกว่าปกติ 1 สเตป
    *   ใน Card: ใช้ `p-6` หรือ `p-8` แทนที่จะเป็น `p-4`
    *   ระหว่าง Section: ใช้ `gap-6` หรือ `gap-8`

### 4. Borders, Shadows & Glassmorphism (มิติและความลึก)
*   **กฎเหล็ก:** เงาต้องนุ่มฟุ้ง ไม่แข็งกระด้าง เส้นขอบต้องบางและกลืนไปกับพื้นหลัง
*   **Cards:** `bg-white dark:bg-slate-800 rounded-2xl shadow-[0_2px_10px_-3px_rgba(6,81,237,0.1)] border border-slate-200/60 dark:border-slate-700/50`
*   **Glass Effect (ความเบลอหรูหรา):** ใช้กับ Navbar หรือ Header ให้พื้นหลังโปร่งแสง
    *   Tailwind: `backdrop-blur-md bg-white/70 dark:bg-slate-900/70 sticky top-0 z-50`

## 🧠 PART 2: UX (User Experience) - กฎการสร้างระบบให้ "ฉลาดและลื่นไหล"

### 1. Micro-interactions (การตอบสนองระดับย่อย)
*   **กฎเหล็ก:** ทุกครั้งที่ผู้ใช้เอาเมาส์ไปชี้ (Hover) หรือคลิก ระบบต้องมีการตอบสนองเสมอเพื่อไม่ให้รู้สึกว่าเว็บค้าง
*   **Transitions:** เติมความนุ่มนวลให้ทุกปุ่มและการ์ด
    *   Tailwind: `transition-all duration-300 ease-in-out hover:-translate-y-0.5 hover:shadow-md active:scale-95`

### 2. Empty States (เมื่อไม่มีข้อมูล)
*   **กฎเหล็ก:** ห้ามปล่อยหน้าจอว่างเปล่าโง่ๆ (เช่น ปล่อยตารางโล่งๆ) เมื่อไม่มีข้อมูล ต้องวาด Empty State ที่สวยงามเสมอ
*   **วิธีทำ:**
    *   ใส่ไอคอนหรือภาพประกอบสีอ่อนๆ ตรงกลางจอ
    *   มีข้อความอธิบายสั้นๆ (เช่น "ยังไม่มีข้อมูลผู้เช่าในระบบ")
    *   ต้องมีปุ่ม Call to Action เสมอ (เช่น "+ เพิ่มข้อมูลผู้เช่าคนแรก")

### 3. Forgiving UI & Error Handling (การจัดการข้อผิดพลาด)
*   **กฎเหล็ก:** ห้ามด่าผู้ใช้ด้วย Error สีแดงน่ากลัวๆ ให้คำแนะนำอย่างสุภาพและแก้ปัญหาให้พวกเขา
*   **Form Validation:** ตรวจสอบข้อมูลทันทีที่พิมพ์เสร็จ (Inline validation) ไม่ใช่รอให้กด Submit แล้วค่อยเด้งเตือน
*   **Destructive Actions:** การลบข้อมูลสำคัญ ต้องมี Confirm Modal เสมอ และปุ่มลบต้องสื่อชัดเจน (เช่น `bg-red-50 text-red-600 hover:bg-red-100`)

### 4. skeleton Loaders (การโหลดที่ดูไม่น่าเบื่อ)
*   **กฎเหล็ก:** ห้ามใช้ Spinner (วงล้อหมุนๆ) ตรงกลางจอขาวๆ ให้ใช้ Skeleton Screen (โครงร่างหน้าจอ) เพื่อให้ผู้ใช้รู้สึกว่าระบบโหลดเร็วกว่าความเป็นจริง
*   Tailwind: `animate-pulse bg-slate-200 dark:bg-slate-700 rounded-lg`

### 5. Progressive Disclosure (เปิดเผยข้อมูลเมื่อจำเป็น)
*   **กฎเหล็ก:** อย่าสาดข้อมูลทุกอย่างใส่หน้าผู้ใช้ในวินาทีแรก
*   **วิธีทำ:** ข้อมูลที่ลึกเกินไป ให้ซ่อนไว้ใน `Accordion`, `Tabs`, หรือคลิก `View Details` เพื่อเปิด Modal ข้อมูลขึ้นมาดู

## 🛠️ PART 3: AI Output Instructions (ข้อบังคับสำหรับ AI Agent)

เมื่อคุณ (AI) ได้รับคำสั่งให้สร้างหรือปรับปรุง UI Component ใดๆ จงปฏิบัติตามกฎดังต่อไปนี้อย่างเคร่งครัด:

1.  **Mobile-First Always:** เขียนคลาส Tailwind ให้รองรับจอมือถือเป็น Default ก่อนเสมอ แล้วค่อยขยาย Layout ด้วย `md:` หรือ `lg:`
2.  **Dark Mode Required:** บังคับใส่คลาส `dark:` ควบคู่กับสีพื้นฐานเสมอ ห้ามลืมเด็ดขาด
3.  **Modern Rounding:** หลีกเลี่ยงมุมเหลี่ยมแข็งกระด้าง ให้ใช้ `rounded-xl` หรือ `rounded-2xl` เป็นมาตรฐานสำหรับ Card, Modal และ Input
4.  **No Pure Black:** ห้ามใช้ `text-black` หรือ `bg-black` ให้ใช้ตระกูล `slate-900` หรือ `gray-900` เพื่อความสบายตา
5.  **Iconography:** เมื่อต้องใช้ไอคอน ให้เลือกใช้ไอคอนสไตล์เส้น (Line Icons) เช่น `Lucide-react` หรือ `Heroicons` ให้มีขนาดพอดีกับข้อความ (มักจะใช้ `w-5 h-5` หรือ `w-4 h-4`)

## 📱 PART 4: Responsive & Omnichannel UX/UI (Desktop & Mobile)

แอปพลิเคชันระดับพรีเมียมต้องมอบประสบการณ์ที่ไร้รอยต่อ (Seamless) ไม่ว่าจะเปิดบนจอขนาด 4 นิ้ว หรือ 27 นิ้ว โปรดใช้กฎเกณฑ์ต่อไปนี้ในการออกแบบ Component ให้รองรับทุกอุปกรณ์

### 1. Mobile-First Layout & Grid (โครงสร้างพื้นฐาน)
* **กฎเหล็ก:** ห้ามเขียนโค้ดโดยอิงจากจอ Desktop ก่อน ให้คิดเสมอว่าหน้าจอมือถือคือค่าเริ่มต้น (Default) แล้วค่อยใช้ Breakpoints ขยายขนาดเมื่อจอใหญ่ขึ้น
* **Technical Rule:** * กำหนดคอลัมน์เริ่มต้นให้เป็น 1 คอลัมน์เสมอ (`grid-cols-1` หรือ `w-full`)
    * ใช้ `md:` (แท็บเล็ต) และ `lg:` (คอมพิวเตอร์) เพื่อแตกเป็น 2 หรือ 3 คอลัมน์ (เช่น `grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3`)

### 2. Navigation Shifting (การปรับเปลี่ยนเมนูนำทาง)
* **กฎเหล็ก:** พื้นที่บนมือถือมีจำกัด ห้ามแสดงเมนูนำทางทั้งหมดกางทิ้งไว้บนจอมือถือ
* **Desktop:** ใช้ Sidebar ด้านข้าง หรือ Navbar ด้านบนที่แสดงลิงก์ครบถ้วน
* **Mobile:** ซ่อนเมนูไว้ใน Hamburger Menu (☰) หรือเปลี่ยนไปใช้ Bottom Navigation Bar (แถบเมนูด้านล่างสุดแบบแอปมือถือ) เพื่อให้นิ้วโป้งกดง่ายที่สุด

### 3. Touch vs. Mouse Interactions (การสัมผัส และ การคลิก)
* **กฎเหล็ก:** บนมือถือไม่มี "Hover" (การเอาเมาส์ชี้) ดังนั้นอย่าซ่อนปุ่มหรือข้อมูลสำคัญไว้ในสถานะ Hover เด็ดขาด
* **Touch Targets:** พื้นที่กดปุ่มบนมือถือ (ปุ่ม, Checkbox, Radio) ต้องมีขนาดกว้างและสูงไม่ต่ำกว่า `44px` ถึง `48px` เสมอ เพื่อไม่ให้นิ้วกดพลาด
* **Active States:** ใส่สถานะเมื่อถูก "ทัช/แตะ" ให้ชัดเจนเพื่อแทนที่ Hover
    * Tailwind: ใช้คลาส `active:scale-95 active:bg-slate-100` เพื่อให้ปุ่มยุบตัวลงเวลานิ้วแตะลงไป

### 4. Smart Forms & Keyboards (ความฉลาดของฟอร์มและแป้นพิมพ์)
* **กฎเหล็ก:** เวลาผู้ใช้กรอกตัวเลขบนมือถือ (เช่น ค่าเช่าห้อง, หน่วยมิเตอร์ไฟ) แป้นพิมพ์ตัวเลขต้องเด้งขึ้นมาทันที ห้ามให้ผู้ใช้ต้องมากดสลับแป้นภาษาเอง
* **Technical Rule:**
    * ฟิลด์ตัวเลข/จำนวนเงิน: ใส่ `inputmode="decimal"` หรือ `type="number"` เสมอ
    * ฟิลด์เบอร์โทรศัพท์: ใส่ `type="tel"` 
    * ฟิลด์อีเมล: ใส่ `type="email"`
    * ห้ามให้กล้องมือถือซูมหน้าจออัตโนมัติเวลาแตะช่อง Input (แก้ด้วยการตั้งฟอนต์ Input ให้มีขนาด `text-base` หรือ 16px ขึ้นไปบนมือถือ)

---

### 5. Data Tables Transformation (การจัดการตารางข้อมูล)
* **กฎเหล็ก:** ตารางข้อมูล (Data Table) ที่มีหลายคอลัมน์คือฝันร้ายของคนใช้มือถือ ห้ามบังคับให้ผู้ใช้ต้องปัดซ้าย-ขวา (Horizontal scroll) อย่างยากลำบาก
* **Desktop:** แสดงผลเป็นตาราง `<table>` ตามปกติเพื่อความชัดเจน
* **Mobile:** เขียนโค้ดสลับให้ตารางแตกออกเป็น "การ์ดข้อมูลรายบุคคล (Card-based List)" เมื่ออยู่บนจอเล็ก (`max-w-sm`) โดยซ่อนหัวตารางและนำ Label มาไว้ด้านหน้าข้อมูลแทน

### 6. Safe Areas & Notches (พื้นที่ปลอดภัยบนสมาร์ทโฟน)
* **กฎเหล็ก:** ระวังขอบจอมือถือ, รอยบากกล้อง (Notch), และเส้น Home Indicator ด้านล่างของ iPhone ดัน UI จนพัง
* **Technical Rule:** ใช้คลาสของ Tailwind ที่รองรับ Safe Area เช่น `pb-safe` หรือการเผื่อ Padding ด้านล่าง `pb-20` ไว้เสมอเมื่อมีแถบเมนูด้านล่าง
